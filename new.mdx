#### Interpreting the Outcome

Every transaction receives a standardized decision outcome. Your backend should map these outcomes to appropriate actions.

| Outcome | Risk Level | Score | Recommended Action |
|---------|------------|-------|--------------------|
| `APPROVE` | `LOW` | 0-29 | Proceed with the transaction immediately. |
| `REVIEW` | `MEDIUM` | 30-59 | Proceed, but flag the transaction for compliance review. |
| `ESCALATE` | `HIGH` | 60-79 | Suspend transaction pending manual officer approval. |
| `BLOCK` | `CRITICAL` | 80-100 | Block the transaction immediately and freeze the account. |

## The Screening Pipeline

When you submit a transaction, Ratel Security runs it through an ultra-low latency, multi-engine pipeline:

### 1. Mandatory Regulatory Engine
These 7 rules implement national AML/CFT Act requirements (e.g. CBN). They are **always active** and **cannot be disabled**.

| Rule | Trigger | Auto-Actions |
|------|---------|-------------|
| **AML-001** Cash Threshold | Cash deposit/withdrawal â‰Ą â‚¦5,000,000 | Generate CTR, Notify Officer |
| **AML-002** Transfer Threshold | Electronic transfer â‰Ą â‚¦10,000,000 | Generate CTR, Notify Officer |
| **AML-003** Structuring | Single txn â‰Ą 80% of threshold, OR 3+ txns in 24h exceeding threshold | Generate SAR, Create Case |
| **AML-004** PEP Screening | Sender identified as Politically Exposed Person | Enhanced Due Diligence |
| **AML-005** Foreign Transfer | International transfer or receiver country â‰  NG | Generate FTR |
| **AML-006** Dormant Account | No transaction activity for â‰Ą 180 days | Create Case |
| **AML-007** Rapid Succession | â‰Ą 5 transactions within 60 minutes from same account | Create Case |

### 2. Global Sanctions & PEP Screening
Cross-references sender and receiver against **1.8 million entities** (UN Security Council, US OFAC, EU Consolidated, UK HMT, Interpol, FATF). 
Our proprietary engine uses multi-algorithm entity resolution to prevent false negatives from typos:
* **Jaro-Winkler** (Fuzzy string similarity)
* **Phonetic Match** (Sound-alike transliterations)
* **Fingerprint Levenshtein** (Edit distance on normalized name tokens)

### 3. Identity Risk Verification
Validates the sender's real-time identity status securely using Tier-1 Identity Partners. If a user's National ID or biometric verification is flagged as `REJECTED` or `EXPIRED`, the transaction is automatically escalated.

### 4. Behavioral Analysis
Builds risk profiles over time by tracking device fingerprints, IP addresses, geographic patterns, and transaction velocity.
* **Multiple Devices:** Detects >3 devices on one account (Account Takeover).
* **Multiple IPs/Proxies:** Detects >5 IPs or known VPN nodes on one account.
* **Multi-Country Activity:** Detects impossible travel or geographic anomalies.

## Scoring Algorithm

Each screening category produces a risk score (0-100) and is assigned a mathematical weight:

| Category | Weight | Rationale |
|----------|--------|-----------|
| **Regulatory Compliance** | **1.5Ă—** | Core compliance is non-negotiable â€” highest authority. |
| **Identity Verification** | **1.3Ă—** | KYC integrity is compliance-critical. |
| **Custom Rules** | **1.2Ă—** | Bank-specific logic is elevated above generic engines. |
| **Sanctions & PEP** | **1.0Ă—** | Standard weight â€” severity captured via outcome escalation. |
| **Behavioral Analysis** | **0.8Ă—** | Supplementary supporting evidence. |

#### Score Floor Guarantee
To prevent critical matches (like a Sanctions hit) from being mathematically diluted by clean engines, the system enforces a strict floor:
`Final Score = MAX(weighted_average, max_engine_score Ă— 0.8)`

**The highest-severity outcome from ANY engine becomes the final decision.** If the Behavioral Analysis says `APPROVE` but Sanctions Screening says `BLOCK`, the final outcome is `BLOCK`.

## Asynchronous Compliance Actions

Ratel Security is designed to decouple heavy compliance tasks from the synchronous request/response cycle. While your API call returns in under 200ms, the platform automatically queues the following actions in the background:

- **Case Management**: Auto-creates compliance cases for any transaction marked `REVIEW`, `ESCALATE`, or `BLOCK`.
- **Regulatory Reporting**: Auto-generates XML drafts for Suspicious Activity Reports (SAR), Currency Transaction Reports (CTR), and Foreign Transaction Reports (FTR).
- **Audit Logging**: Maintains a cryptographic, immutable audit log of the screening event for regulatory inspection.
- **Alert Dispatch**: Sends email and dashboard notifications to your Compliance Officers.

## Asynchronous Webhooks

Ratel Security can send real-time asynchronous updates to your backend whenever a transaction's status changes due to a compliance officer's manual review.

<Note>
**Configure your Webhook:** You must configure your `Webhook URL` and `Webhook Secret` in the **Developer Webhooks** section of the API Keys page in your Dashboard.
</Note>

### Webhook Payload

When an officer resolves a case (e.g., marking it as `RESOLVED_TRUE_POSITIVE` or `RESOLVED_FALSE_POSITIVE`), Ratel Security will send a `POST` request to your configured URL:

<CodeGroup>
```json Example Payload
{
  "event": "case.resolved",
  "timestamp": "2026-05-08T15:00:00Z",
  "data": {
    "caseId": "case_12345abcde",
    "status": "RESOLVED_FALSE_POSITIVE",
    "resolutionNote": "Customer provided valid source of funds documentation.",
    "relatedTransactionId": "TXN-2026-001",
    "updatedAt": "2026-05-08T15:00:00Z"
  }
}
```
</CodeGroup>

### Security & Signature Verification

To ensure webhooks are genuinely originating from Ratel Security, every request includes an `X-Ratel-Signature` header. This is an HMAC-SHA256 hash of the JSON payload signed using your `Webhook Secret`.

You should compute the hash of the raw request body on your server and compare it to the header to prevent spoofing attacks.

```javascript Node.js Verification
const crypto = require('crypto');

function verifyWebhook(reqBody, signatureHeader, secret) {
  const hash = crypto
    .createHmac('sha256', secret)
    .update(JSON.stringify(reqBody))
    .digest('hex');
    
  return `sha256=${hash}` === signatureHeader;
}
```

## Error Handling

Ratel Security uses standard HTTP response codes to indicate the success or failure of an API request.

| Code | Meaning | Solution |
|------|---------|----------|
| `400` | Bad Request | Check the validation errors in the response body. Ensure all required fields are present. |
| `401` | Unauthorized | Your `X-API-Key` header is missing, invalid, or your workspace has been suspended. |
| `429` | Too Many Requests | You have exceeded your tenant's rate limit. |
| `500` | Internal Server Error | We had a problem with our server. Try again later. |

<Note>
**Fail-Open Policy:** If the Sanctions or Behavioral engines experience upstream network downtime, Ratel Security implements a "Fail-Open" policy. This ensures your legitimate transactions continue to process without interruption, while logging the partial screening for retroactive review.
</Note>
