## Business Transactions

When screening transactions from corporate entities, include the `metadata` object to trigger KYB (Know Your Business) verification:

```json
{
  "externalId": "TXN-2026-002",
  "type": "TRANSFER",
  "channel": "INTERNET_BANKING",
  "amount": 25000000,
  "currency": "NGN",
  "senderAccountNumber": "0123456789",
  "senderName": "Acme Nigeria Ltd",
  "receiverName": "Global Shipping Co",
  "metadata": {
    "entityType": "BUSINESS",
    "senderRcNumber": "RC123456"
  }
}
```

### Metadata Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `metadata.entityType` | `string` | | `"BUSINESS"` to trigger KYB screening. `"INDIVIDUAL"` or absent skips KYB. |
| `metadata.senderRcNumber` | `string` | | CAC registration number. Also triggers KYB screening if provided. |

### KYB Risk Impact

When `entityType` is `"BUSINESS"` or `senderRcNumber` is present, the pipeline adds a **KYB Verification** layer:

| KYB Status | Score | Outcome |
|------------|-------|---------|
| **APPROVED** | 0 | `APPROVE` |
| **BOS_VERIFIED** | 10 | `APPROVE` |
| **PENDING / CAC_VERIFIED / DIRECTORS_VERIFIED** | 40 | `REVIEW` |
| **NONE** (no KYB record) | 60 | `REVIEW` |
| **REJECTED** | 85 | `BLOCK` |

> **Tip:** For the best experience, ensure the business has completed KYB verification (CAC lookup ã director verification ã beneficial owner screening ã approval) before they begin transacting.

## How Screening Works

Every transaction passes through multiple screening layers in sequence. The entire pipeline completes in **80ã200ms** (p95).

### The Pipeline

| Stage | Engine | Latency | What It Does |
| ----- | ------------------------- | ------- | ----------------------------------------------------------- |
| 1 | **Validation** | ~5ms | Schema validation, idempotency check, persist transaction |
| 2 | **Context Enrichment** | ~15ms | Fetch sender's recent history, account dormancy, KYC/KYB status |
| 3 | **Sanctions & PEP** | ~25ms | Screen names against global watchlist entities |
| 4 | **Regulatory Compliance** | ~2ms | Execute 7 mandatory AML rules |
| 5 | **Identity Risk (KYC)** | ~1ms | Evaluate sender's KYC verification status |
| 6 | **Identity Risk (KYB)** | ~1ms | For business transactions ã evaluate KYB verification status |
| 7 | **Behavioral Analysis** | ~50ms | ML scoring for device, velocity, and geographic anomalies |

### Mandatory Regulatory Rules

These 7 rules are **always active** and **cannot be disabled**. They implement national AML/CFT Act requirements.

| Rule | Trigger | Auto-Actions |
| ------------------------------ | -------------------------------------------------------------------- | ---------------------------- |
| **AML-001** Cash Threshold | Cash ãË ãÎ5,000,000 | Generate CTR, Notify Officer |
| **AML-002** Transfer Threshold | Transfer ãË ãÎ10,000,000 | Generate CTR, Notify Officer |
| **AML-003** Structuring | Single txn ãË 80% of threshold, OR 3+ txns in 24h exceeding threshold | Generate SAR, Create Case |
| **AML-004** PEP Screening | Sender is a Politically Exposed Person | Enhanced Due Diligence |
| **AML-005** Foreign Transfer | International transfer or receiver country ã  NG | Generate FTR |
| **AML-006** Dormant Account | No activity for ãË 180 days | Create Case |
| **AML-007** Rapid Succession | ãË 5 txns within 60 minutes from same account | Create Case |

### Sanctions & PEP Matching

Sender and receiver names are cross-referenced against **millions of entities** from:

- UN Security Council Consolidated List
- US OFAC SDN + Consolidated Lists
- UK HMT Sanctions List
- Interpol Red Notices
- Politically Exposed Persons from 100+ countries
- FATF Blacklist / Greylist entities

<Note>
  **EU Consolidated Financial Sanctions** is temporarily excluded from the default
  configuration. Contact support to enable it with an `EU_CSL_TOKEN`.
</Note>

The engine uses multi-algorithm entity resolution to catch typos, transliterations, and aliases:

| Algorithm | Purpose |
| --------------------------- | ----------------------------------------------- |
| **Exact Match** | Case-normalized literal comparison |
| **Jaro-Winkler** | Fuzzy string similarity (handles typos) |
| **Phonetic Match** | Sound-alike matching (handles transliterations) |
| **Fingerprint Levenshtein** | Edit distance on normalized name tokens |

Only matches with ãË 70% composite confidence are flagged.

### Behavioral Analysis

| Signal | Trigger | Detects |
| -------------------- | ---------------------------- | --------------------------------------- |
| **Multiple Devices** | > 3 devices on one account | Account takeover, credential sharing |
| **Multiple IPs** | > 5 IPs on one account | Proxy/VPN usage, bot activity |
| **Multi-Country** | > 2 countries on one account | Impossible travel, geographic anomalies |

### Scoring Algorithm

Each engine produces a risk score (0ã100). Scores are aggregated using weighted severity:

| Category | Weight | Rationale |
| ------------------------- | -------- | -------------------------------------------------------- |
| **Regulatory Compliance** | **1.5û** | Core compliance ã highest authority |
| **Identity Verification** | **1.3û** | KYC/KYB integrity is compliance-critical |
| **Custom Rules** | **1.2û** | institution-specific logic elevated above generic engines |
| **Sanctions & PEP** | **1.0û** | Standard weight ã severity via outcome escalation |
| **Behavioral Analysis** | **0.8û** | Supplementary evidence |

**Score Floor Guarantee:** `Final Score = MAX(weighted_average, max_engine_score û 0.8)`

This prevents a critical match (e.g., Sanctions hit scoring 95) from being diluted by clean engines. The **highest-severity outcome from any engine becomes the final decision.**

### Asynchronous Side-Effects

While the API responds in under 200ms, the following actions are queued in the background:

- **Case Management** ã Auto-creates compliance cases for `REVIEW`, `ESCALATE`, or `BLOCK` outcomes
- **Regulatory Reporting** ã Auto-generates XML drafts for SAR, CTR, and FTR reports
- **Audit Logging** ã Immutable audit trail for regulatory inspection
- **Alert Dispatch** ã Email and dashboard notifications to Compliance Officers

## Webhooks

Ratel Security sends real-time notifications to your backend when a compliance officer resolves a case linked to one of your transactions.

<Note>
  **Setup:** Configure your `Webhook URL` and `Webhook Secret` in the
  **Developer Webhooks** section of the API Keys page in your Dashboard.
</Note>

### Events

| Event | Trigger |
| --------------- | ----------------------------------------------------------------------------------------------------- |
| `case.resolved` | A compliance officer marks a case as `RESOLVED_TRUE_POSITIVE`, `RESOLVED_FALSE_POSITIVE`, or `CLOSED` |

### Payload

<CodeGroup>
```json case.resolved
{
  "event": "case.resolved",
  "timestamp": "2026-05-08T15:00:00Z",
  "data": {
    "caseId": "case_12345abcde",
    "status": "RESOLVED_FALSE_POSITIVE",
    "resolutionNote": "Customer provided valid source of funds documentation.",
    "relatedTransactionId": "TXN-2026-001",
    "updatedAt": "2026-05-08T15:00:00Z"
  }
}
```
</CodeGroup>

**Handling the resolution in your system:**

| `status` Value | Meaning | Your Action |
| ------------------------- | ----------------------------------------------- | --------------------------------------- |
| `RESOLVED_FALSE_POSITIVE` | Officer confirmed the transaction is legitimate | Unfreeze the transaction and process it |
| `RESOLVED_TRUE_POSITIVE` | Officer confirmed suspicious activity | Permanently block and escalate internally |
| `CLOSED` | Case closed after investigation | Take no further action |

### Delivery Guarantees

Webhook delivery is backed by a persistent message queue with automatic retries:

| Property | Value |
| --------------------------- | ---------------------------------------- |
| **Max Attempts** | 5 |
| **Backoff Strategy** | Exponential (5s ã 10s ã 20s ã 40s ã 80s) |
| **Request Timeout** | 10 seconds per attempt |
| **Completed Job Retention** | 24 hours |
| **Failed Job Retention** | 7 days |

<Warning>
  **Your endpoint must return a `2xx` status code.** Any non-2xx response will
  trigger automatic retries. If all 5 attempts fail, the event is retained in
  the dead-letter queue for 7 days.
</Warning>

### Signature Verification

Every webhook includes an `X-Ratel-Signature` header ã an HMAC-SHA256 hash of the raw JSON body, signed with your Webhook Secret. **Always verify this server-side.**

<CodeGroup>
```javascript Node.js (Express)
const crypto = require('crypto');

app.post('/ratel/webhooks', express.raw({ type: 'application/json' }), (req, res) => {
const signature = req.headers['x-ratel-signature'];
const hash = crypto
.createHmac('sha256', process.env.RATEL_WEBHOOK_SECRET)
.update(req.body)
.digest('hex');

if (`sha256=${hash}` !== signature) {
return res.status(401).send('Invalid signature');
}

const event = JSON.parse(req.body);

if (event.event === 'case.resolved') {
if (event.data.status === 'RESOLVED_FALSE_POSITIVE') {
// Unfreeze the transaction in your system
} else if (event.data.status === 'RESOLVED_TRUE_POSITIVE') {
// Permanently block and escalate
}
}

res.status(200).send('OK');
});

````

```python Python (Flask)
import hmac
import hashlib
import json
from flask import Flask, request, abort

app = Flask(__name__)

@app.route('/ratel/webhooks', methods=['POST'])
def handle_webhook():
    signature = request.headers.get('X-Ratel-Signature', '')
    expected = 'sha256=' + hmac.new(
        RATEL_WEBHOOK_SECRET.encode(),
        request.data,
        hashlib.sha256
    ).hexdigest()

    if not hmac.compare_digest(signature, expected):
        abort(401)

    event = json.loads(request.data)

    if event['event'] == 'case.resolved':
        if event['data']['status'] == 'RESOLVED_FALSE_POSITIVE':
            # Unfreeze the transaction
            pass
        elif event['data']['status'] == 'RESOLVED_TRUE_POSITIVE':
            # Permanently block
            pass

    return 'OK', 200
````

## Error Handling

Standard HTTP status codes indicate success or failure:

| Code | Meaning | Solution |
| ----- | --------------------- | -------------------------------------------------------------------------------------------- |
| `200` | Success | Request processed successfully. |
| `400` | Bad Request | Check validation errors in the response body. Ensure all required fields are present. |
| `401` | Unauthorized | Your `X-API-Key` is missing, invalid, or revoked. Check the API Keys page in your Dashboard. |
| `409` | Conflict | The `externalId` was already processed. The cached verdict is returned. |
| `429` | Too Many Requests | You exceeded your workspace's rate limit. Implement exponential backoff. |
| `500` | Internal Server Error | Unexpected server error. Retry with backoff. |

<Note>
  **Fail-Open Policy:** If the Sanctions or Behavioral engines experience
  upstream downtime, Ratel Security implements a fail-open policy ã your
  legitimate transactions continue processing without interruption, while
  partial screenings are logged for retroactive review.
</Note>
